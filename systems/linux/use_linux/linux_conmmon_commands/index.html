find命令

$ find . -type f -size +500M  	# 查询当前超过500M的文件
$ find . -type f -exec ls -l {} \;  	# 查询当前文件，并以长格式列出
$ find . -type f -exec chmod 644 {} \;   # f:代表文件，设置644权限
$ find . -type d -exec chmod 755 {} \;   # d代表目录，设置755权限
$ find ./* -type f -mtime +15 | xargs ls -lt  	# 过滤出来文件按时间排序
$ find . -type f -mtime -30 -exec gzip -c ./* >/tmp/nginx-wwwlogs-09-27.gz {} \;  	# atime：最后一次访问时间，ctime：最后一次改变权限时间，mtime：最后一次修改时间
$ find . -name "*.log" -exec cp {} test \;  	# 查找当前目录下以.log结尾文件，
													# 拷贝并且新文件重新命名为：test

$ find . -name "*.log" -mtime +5 -ok rm {} \;  	# 和-exec的作用相同，
	# 只不过以一种更为安全的模式来执行该参数所给出的shell命令，
	# 在执行每一个命令之前，都会给出提示，让用户来确定是否执行

$ find . -name "*.log" -mtime +6 -exec du -sh {} \;  	# 查看当前目录下以.log结尾并超过七天的文件大小
$ find /etc -name "passwd*" -exec grep "root" {} \;

# 查找并且打包
$ find /your/directory/path -name "*.tar" -exec tar -xvf {} \;
# 查找并打包到指定目录
$ find /your/directory/path -name "*.tar" -exec tar -xvf {} -C /path/to/extracted/directory \;

  • TCP传输控制协议

  • SYN同步包，用于建立连接

    功能: 用于建立连接。SYN包是TCP三次握手过程中的第一步

    使用场景: 客户端向服务器发送SYN包以开始连接的建立

  • SYN-ACK同步-确认包，响应 SYN包以确认建立连接

    功能: 用于响应SYN包，并确认接收到的SYN包，表示服务器准备建立连接

    使用场景: 服务器向客户端回复SYN-ACK包，作为三次握手的第二步

  • ACK确认包，用于确认数据包的接收

    功能: 用于确认接收到的数据包。ACK包是TCP协议中的确认机制

    使用场景: 客户端收到服务器的SYN-ACK包后，发送ACK包以完成三次握手，建立连接

  • FIN结束包，用于结束连接

    功能: 用于结束连接。FIN包是 TCP`四次挥手中的一部分

    使用场景: 连接的一方发送FIN包，表示该端已经完成数据发送并希望关闭连接

  • RST重置包，用于重置连接

    功能: 用于强制重置连接，立即终止连接。RST包用于通知对方连接发生异常，需要立即关闭

    使用场景: 当发生错误或不需要的连接时使用，如端口未开放、应用异常等

  • PSH推送包，用于推送数据，要求接收方立即处理

    功能: 用于通知接收方立即处理数据，而不是等待缓冲区填满

    使用场景: 发送方希望接收方立即处理数据，通常用于实时应用

  • URG紧急包，指示紧急数据

    功能: 指示数据中的紧急信息，并通知接收方优先处理这些数据

    使用场景: 数据包中包含紧急数据，要求接收方立即处理

  • UDP用户数据报协议

  • 数据报Datagram

    功能: 这是UDP协议中最基本的包类型，包含了数据的完整内容和必要的头部信息

    组成部分:

    • 头部:
      • 源端口号（16 位）
      • 目的端口号（16 位）
      • 长度（16 位）：包括头部和数据的总长度
      • 校验和（16 位）：用于检测数据包在传输过程中是否出现错误
    • 数据: 实际传输的数据内容

  • 空数据报Empty Datagram

    功能: 一个没有数据负载的 UDP 数据报，主要用于保持连接的状态或作为信号量。

    场景: 例如，心跳检测或网络测试等场景中，可能会使用空数据报来保持连接或检查网络状态

  • UDP协议特点

    无连接: UDP不建立连接，每个数据报都是独立的

    无序: 数据包的顺序不保证，因此接收方可能收到乱序的数据

    无可靠性: UDP不保证数据包的交付，也没有重传机制。如果数据包在传输过程中丢失，UDP不会进行重发

    无流量控制: UDP不控制数据流量，也不进行流量调节

  • UDP数据报的基本结构

    头部Header:

    • 源端口号: 表示发送方的端口
    • 目的端口号: 表示接收方的端口
    • 长度: 包括头部和数据的总长度
    • 校验和: 用于检查数据的完整性

  • 数据Data:

    • 实际传输的应用数据
