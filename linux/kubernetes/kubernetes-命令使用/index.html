kubectl资源、命令

    1. kubectl：是kubernetes集群的命令行工具，通过它能够对集群本身进行管理，并能够在集群上进行容器化应用的安装和部署

kubectl api-resources：是查看kubernetes资源命令

    1. Kubernetes中所有的内容都抽象为资源，所有命令都是围绕资源操作、执行
资源分类资源名称缩写资源作用
集群级别资源nodesno集群组成部分
集群级别资源namespacesns隔离Pod
PodsPodspo装载容器
Pod资源控制器replicationcontrollersrc比较原始的Pod控制器，已经被废弃，由ReplicaSet替代
—replicasetsrs保证指定数量的Pod运行，并支持Pod数量变更，镜像版本变更
—deploymentsdeploy通过控制ReplicaSet来控制Pod，并支持滚动升级、版本回退
—daemonsetsds在集群中的指定Node上都运行一个副本，一般用于守护进程类的任务
—jobs—它创建出来的Pod只要完成任务就立即退出，用于执行一次性任务
—cronjobscj它创建的Pod会周期性的执行，用于执行周期性的任务
—horizontalpodautoscalershpa可以根据集群负载自动调整Pod的数量，实现削峰填谷
Pod资源控制器statefulsetssts管理有状态的应用
服务发现资源servicessvc统一Pod对外接口
服务发现资源ingressing统一Pod对外接口

—persistentvolumespv存储
存储资源persistentvolumeclaimspvc存储

基本命令详解

    1. kubectl --help：可以通过--help查看详细的操作命令
基本命令(初学者)命令翻译命令作用
—create创建从文件或标准输入创建资源
—expose暴露将复制控制器、服务、部署或pod暴露为新的Kubernetes服务Service
—run运行在集群中运行一个指定的镜像
基本命令(初学者)set设置设置一个对象的特定特征
    1. kubectl命令的语法如下$ kubectl [command] [type] [name] [flags]
  • 2.1 command: 指定要对资源执行的操作，比如：create、get、delete
  • 2.2 type：指定资源的类型，比如：deployment、pod、service
  • 2.3 name： 指定资源的名称，名称大小写敏感
  • 2.4 flags： 指定额外的可选参数
    1. kubectl create --help：创建命令用于创建资源
  • 3.1 yaml文件详解、使用kubectl create命令生成yaml文件
  • 3.2 deployment：无状态应用部署
  • 3.3 java-demo：自定义名称
  • 3.4 --image：使用镜像
  • 3.5 --dry-run：测试
  • 3.6 -o：输出
  • 3.7 >/tmp/deploy.yaml：定向输出到文件，不加>/tmp/deploy.yaml 输出到屏幕查看
# 创建一个namespace命名空间
$ kubectl create namespace dev

$ kubectl create service clusterip ngx-dep --tcp=80:80  # 命令行创建service

$ kubectl create -f  ns-dev.yaml    # 根据yaml文件创建资源
    1. kubectl expose --help端口暴露
  • 4.1 expose端口设置固定选项
  • 4.2 java-demo： 自定义名称
  • 4.3 --port：Pod的Service服务端口，集群内部互通端口
  • 4.4 --target-port：Pod容器内服务端口端口
  • 4.5 --type：随机生成外部访问端口
  • 4.6 >java-demo-svc.yaml：定向输出到文件，不加>java-demo-svc.yaml输出到屏幕查看

Kubernetes部署命令

部署命令命令翻译命令作用
—rollout首次展示管理资源的发布
—scale规模扩(缩)容Pod的数量,为Deployment、ReplicaSet或Replication Controller设置新的大小
—autoscale自动调整自动调整Pod的数量，自动缩放Deployment、ReplicaSet、StatefulSet或ReplicationController
    1. kubectl rollout --help：管理资源
  • 1.1 status：显示当前升级状态
  • 1.2 history：显示升级历史记录
  • 1.3 pause：暂停版本升级过程
  • 1.4 resume：继续已经暂停的版本升级过程
  • 1.5 restart：重启版本升级过程
  • 1.6 undo：回滚到上一级版本(可以使用--to-revision，指定版本)
  • 1.7 --record：记录CHANGE-CAUSE(改变原因)操作记录,便于回滚
# 升级
$ kubectl apply -f pc-deployment.yml --record  # --record，记录操作记录,便于回滚

# 继续更新过程
$ kubectl rollout resume deployment nginx-deployment -n app

# 相当于灰度发布
$ kubectl rollout pause deployment nginx-deployment -n app

# 通过修改deployment文件更改
$ kubectl apply -f nginx-deployment.yml --record
$ kubectl rollout pause deployment nginx-deployment -n app  # pause 暂定更改

# 查看升级状态
$ kubectl rollout status deployment/nginx-deployment

# 查看升级历史版本
$ kubectl rollout history deployment/web
$ kubectl rollout history deployment -n app
# 回滚
$ kubectl rollout undo deployment/web --revision=2
$ kubectl rollout undo deployment/nginx-deployment --to-revision=1 -n app # 版本回滚到v1
    1. kubectl scale --help：扩缩容Pod的数量
  • 2.1 --replicas：参数定义Pod数量
$ kubectl scale rc redis --replicas=3 -n web    #操作pod 的扩容和缩容
$ kubectl scale --replicas=2 -f redis-slave-deployment.yaml
    1. kubectl autoscale --help：使Deployment设定，使用默认的自动伸缩策略，指定目标CPU使用率，使其Pod数量在2到10之间
$ kubectl autoscale deployment deployment-nginx --cpu-percent=60 --min=2 --max=10

集群管理命令taint污点taint更新一个或多个节点上的污点
    1. kubectl top --help：监控集群内存CPU指标
  • 1.1 metrics-server可以用来收集集群中的资源使用情况、需要单独安装
  • 1.2 Kubernetes指标服务器 、 安装Prometheus监控服务器资源
  • 1.3 新增参数：hostNetwork: true，用于使用宿主机网络
  • 1.4 国内修改为阿里云地址：image: registry.cn-hangzhou.aliyuncs.com/google_containers/metrics-server-amd64:v0.3.6
  • 1.5 新增参数：- --kubelet-insecure-tls，不验证客户端证书，仅用于测试目的，kubelet的10250端口使用的是https协议，连接需要验证tls证书
$ wget https://github.com/kubernetes-sigs/metrics-server/releases/download/v0.6.1/components.yaml
$ sudo cat >>components.yaml<<-EOF  # 具体添加一下内容
apiVersion: apps/v1
kind: Deployment
metadata:
	labels:
		k8s-app: metrics-server
	name: metrics-server
	namespace: kube-system
spec:
	selector:
		matchLabels:
			k8s-app: metrics-server
	strategy:
		rollingUpdate:
			maxUnavailable: 0
	template:
		metadata:
			labels:
				k8s-app: metrics-server
		spec:
			hostNetwork: true  	# 新增：hostNetwork: true
			containers:
				- args:
					- --cert-dir=/tmp
						- --secure-port=4443
						- --kubelet-preferred-address-types=InternalIP,Hostname,InternalDNS,ExternalDNS,ExternalIP
						- --kubelet-use-node-status-port
						- --metric-resolution=15s
						- --kubelet-insecure-tls  	# 新增：- --kubelet-insecure-tls
						image: k8s.gcr.io/metrics-server/metrics-server:v0.6.1
						imagePullPolicy: IfNotPresent
EOF

$ kubectl apply -f components.yaml

$ kubectl top node  --use-protocol-buffers
$ kubectl top  pod/node  -n dev
    1. kubectl taint --help：污点配置
  • 2.1 查看节点上的污点
$ kubectl describe nodes k8s-node1 | grep Taints
  • 2.2 设置污点
  • 2.2.1 tag=webapps、tag：污点名称，可以自定义

  • 2.3 去除污点
  • 2.3.1 -：污点名称后面跟减号就是去除污点意思

$ kubectl taint node k8s-node1 tag-

$ kubectl replace -f rc-nginx.yaml

environment(环境)环境：dev(开发)、qa/test(测试)、gray(灰度)、production(生产)、op(运维)、ui,as,pc,sc(应用类(app))—
tier(层级)层级：frontend(前端)、backend(后端)、cache(缓存)—
partition(分区/模块)分区/模块：customerA(业务A/客户A)、customerB(业务B/客户B)—
version(版本号)版本号：v1.1、v1.2—
# 显示pod的标签
$ kubectl label pod  --show-labels -n web -o wide

# 为pod打上标签
$ kucectl label pod  nginx-pod version=1.0 -n web
$ kubectl label nodes k8s-node-01  zabbix-server=true  	# 给k8s-node-01节点打上标签

# 覆盖原来的标签：--overwrite
$ kucectl label pod  nginx-pod version=1.0  --overwrite -n web

# 删除标签
$ kucectl label pod  nginx-pod version- -n web
